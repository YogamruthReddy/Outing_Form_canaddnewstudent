### What is this system?

The **Outing Form Management System** is a fully browser-based, zero-installation web application that completely replaces paper-based outing registers used in academic hostels and institutions.

**The real problems it solves:**

| ❌ Before | ✅ After |
|---|---|
| Paper registers — lost, torn, messy | Digital records — always safe |
| Manual entry — full of human errors | Validated forms — clean data |
| No admin visibility in real-time | Instant table view for admins |
| Slow approvals — no tracking | Structured records with timestamps |
| No way to add new students easily | One-click dynamic registration |

### ⚙️ Step-by-Step Engine

```
┌─────────────────────────────────────────────────────────────────────┐
│  STEP 01  ──►  User opens index.html in any browser                 │
│                └─► HTML renders the full outing form UI              │
├─────────────────────────────────────────────────────────────────────┤
│  STEP 02  ──►  Student fills in the form                            │
│                ├─► Name, Roll Number, Year, Branch                  │
│                ├─► Outing Date, Time Out, Expected Time In          │
│                └─► Purpose and Destination                          │
├─────────────────────────────────────────────────────────────────────┤
│  STEP 03  ──►  Validation Module fires on Submit click              │
│                ├─► Empty fields       → inline error shown          │
│                ├─► Invalid time range → alert triggered             │
│                └─► All valid          → proceed to store            │
├─────────────────────────────────────────────────────────────────────┤
│  STEP 04  ──►  Storage Module saves the entry                       │
│                └─► JSON.stringify() → localStorage key              │
├─────────────────────────────────────────────────────────────────────┤
│  STEP 05  ──►  Display Module re-renders the records table          │
│                └─► JSON.parse() → maps to <tr> rows dynamically     │
├─────────────────────────────────────────────────────────────────────┤
│  STEP 06  ──►  Admin views all entries in the table panel           │
│                └─► Sorted by timestamp, newest first               │
├─────────────────────────────────────────────────────────────────────┤
│  STEP 07  ──►  Admin uses Add Student module                        │
│                ├─► Fills student registration form                  │
│                └─► Student profile stored, available in system      │
└─────────────────────────────────────────────────────────────────────┘
```

## 📘 Usage Guide

<div align="center">

### 👨‍🎓 Student Flow

| Step | Action | Result |
|:---:|:---|:---|
| **1** | Open `index.html` in browser | Outing form renders instantly |
| **2** | Enter **Name** and **Roll Number** | Fields accept and validate input |
| **3** | Pick **Date**, **Time Out**, **Expected In** | Time range cross-validated |
| **4** | Type **Purpose** and **Destination** | Short text required to proceed |
| **5** | Click **Submit Outing Request** | Record saved + table updates |

<br/>

### 🛠️ Admin Flow

| Step | Action | Result |
|:---:|:---|:---|
| **1** | Open the application | Admin panel loads alongside the form |
| **2** | Scroll to **Records Table** | All submissions shown in order |
| **3** | Click **Add New Student** | Student registration panel opens |
| **4** | Fill profile: Name, Roll, Branch, Year | Input fields validated |
| **5** | Click **Save Student** | Profile stored, available in dropdowns |

| `📝 docs:` | Documentation only changes |
| `🎨 style:` | CSS or UI visual changes |
| `♻️ refactor:` | Code restructured, no feature/fix |
| `⚡ perf:` | Performance improvements |
| `🔒 security:` | Security patches |

- ✅ One feature or fix per pull request

- ✅ Test your change in at least two browsers
